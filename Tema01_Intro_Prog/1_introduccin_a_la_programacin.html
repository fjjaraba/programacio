1. Introducción a la programación

Programa Informático

Un programa es un conjunto de instrucciones encaminadas a resolver y dar solución a un determinado problema. En este curso se van a aprender y dar uso a las características fundamentales de cualquier lenguaje de programación.

Veamos mediante un gráfico los pasos necesarios para la realización de un programa:

Ejecución de un programa

A continuación se muestra una lista de los pasos del proceso:

  1. El SO busca el nombre del programa en el directorio del disco actual. Si no lo encuentra, busca en una lista predeterminada de directorios (llamados rutas). Si tampoco lo encuentra, emite un mensaje de error.
  2. Si se encuentra el archivo del programa, el SO obtiene la información básica sobre el archivo del programa desde el directorio del disco, incluyendo el tamaño del archivo y su ubicación física en la unidad de disco.
  3. El SO, mediante el cargador de programas,  determina una ubicación disponible en la memoria y carga el archivo binario del programa en la memoria. Asigna un bloque de memoria al programa y registra información sobre el tamaño y ubicación en una tabla (a veces llamada tabla de descriptores).
  4. El SO comienza la ejecución de la primera instrucción del programa (su punto de entrada). Tan pronto como el programa comienza a ejecutarse, se le llama proceso.
  5. El SO asigna al proceso un número de identificación (ID de proceso), que se utiliza para realizar un seguimiento mientras se ejecuta.
  6. El proceso se ejecuta por sí mismo. Es responsabilidad del SO realizar un seguimiento de la ejecución del proceso y responder a las solicitudes de recursos del sistema.
  7. Cuando el proceso finaliza, se elimina de la memoria.

A nivel de la CPU, cuando se ejecuta un programa, lo que sucede es lo siguiente:

  1. Fetch: el ciclo de ejecución comienza con la obtención de una instrucción desde la memoria principal o RAM. La instrucción en el contador de programa actual (PC) se obtendrá y se almacenará en el registro de instrucciones (IR) para que la CPU sepa dónde localizar esta primera instrucción que dará comienzo al programa.
  2. Decodificación: durante este ciclo, la instrucción codificada presente en el registro de instrucciones (IR) es interpretada por el decodificador. Es decir, la unidad de control, a través del microcódigo que tiene, traducirá la instrucción en microoperaciones o señales que determinarán qué es lo que las unidades funcionales o de ejecución deben hacer. Por ejemplo, si se trata de una instrucción suma se le envía una señal a la ALU para que se ponga en modo suma.
  3. Ejecución: la Unidad Aritmético Lógica (ALU) es donde se realizan las operaciones entre dos operandos indicados por el operador en las instrucciones. Por ejemplo, si la instrucción es sumar dos números, aquí se llevará a cabo la suma. La ALU toma dos valores y produce uno de salida, que es el resultado de la operación.
  4. Acceso a la memoria: hay solo dos tipos de instrucciones que acceden a la memoria: LOAD (cargar) y STORE (almacenar). LOAD copia un valor desde la memoria a un registro, mientras que STORE copia un valor de un registro a la memoria. Otras instrucciones omiten este paso.
  5. Actualizar el archivo de registros: en este paso, el resultado/salida de la ALU se escribe de vuelta en el archivo de registros para actualizarlo. El resultado también puede ser el resultado de una carga desde la memoria. Algunas instrucciones no tienen resultados para almacenar. Por ejemplo, las instrucciones BRANCH (salto) y JUMP (salto incondicional) no tienen resultados para almacenar.
  6. Actualizar el contador de programa (PC): finalmente, al finalizar la ejecución de la instrucción actual, debemos actualizar el registro contador de programa (PC) a la dirección de la próxima instrucción, de modo que podamos volver al paso 1 donde la CPU obtendrá la siguiente instrucción. Esto se consigue sumando +1 a la dirección actual del PC, hasta terminar la secuencia de instrucciones del programa. Sin embargo, el contador de programa puede necesitar establecerse en una dirección de memoria diferente a la siguiente si la instrucción fue una instrucción BRANCH o JUMP, aunque no entraré en esto para no complicarlo todo más…

A nivel interno, la ejecución de un programa en un ordenador implica una serie de procesos y componentes clave que trabajan juntos para cargar, interpretar y ejecutar el código del programa. Aquí tienes una descripción más detallada de cómo se ejecuta un programa a nivel interno:

  1. Procesador (CPU - Central Processing Unit)
    • El programa se carga en la memoria RAM desde el dispositivo de almacenamiento (disco duro, SSD, etc.).
      • La CPU  ejecuta las instrucciones del programa.
  2. Compilación o Interpretación (dependiendo del lenguaje)
    • Si el programa está escrito en un lenguaje compilado, como C++ o Java, primero debe compilarse en un código ejecutable antes de su ejecución.
    • Si el programa está escrito en un lenguaje interpretado, como Python o JavaScript, se interpreta línea por línea durante la ejecución.
  3. Sistema Operativo
    • El SO actúa como intermediario entre el programa y el hardware del ordenador.
    • Asigna recursos, como tiempo de CPU y memoria, al programa en ejecución.
  4. Gestión de Memoria
    • El sistema operativo asigna espacio en la memoria RAM para almacenar el código del programa y sus datos.
    • Durante la ejecución, el programa accede a la memoria para cargar y almacenar datos temporales.
  5. Instrucciones y Datos:
    • Las instrucciones del programa y los datos necesarios se cargan en la memoria RAM para su procesamiento.
    • La CPU busca y ejecuta las instrucciones una por una.
  6. Ejecución
    • La CPU ejecuta las instrucciones del programa secuencialmente, realizando operaciones aritméticas, lógicas y de almacenamiento según sea necesario.
    • Los datos se leen, procesan y actualizan según lo requieran las instrucciones.
  7. Gestión de Eventos
    • El SO y la aplicación manejan eventos del usuario, como clics de ratón y pulsaciones de teclas.
  8. Finalización del Programa
    • Cuando el programa ha completado su tarea o se ha cerrado manualmente, el SO libera los recursos asignados al programa, como memoria y CPU.
    • Los procesos asociados al programa se terminan, liberando los recursos.
  9. Liberación de Recursos
    • La memoria asignada al programa se libera para otros usos.
    • Los archivos temporales y otros recursos utilizados por el programa pueden ser eliminados o gestionados por el SO.

Representación del Flujo de Datos

Un Diagrama de Flujo de Datos ( DFD )  traza el flujo de la información para cualquier proceso o sistema. Con el DFD se puede representar el futuro funcionamiento del programa a desarrollar.

Emplea símbolos definidos, como rectángulos, círculos y flechas, además de etiquetas de texto breves, para mostrar las entradas y salidas de datos, los puntos de almacenamiento y las rutas entre cada destino. Los diagramas de flujo de datos pueden variar desde simples panoramas de procesos incluso trazados a mano, hasta DFD muy detallados y con múltiples niveles que profundizan progresivamente en cómo se manejan los datos. Se pueden usar para analizar un sistema existente o para modelar uno nuevo. De forma similar a todos los mejores diagramas y gráficos, un DFD puede con frecuencia "decir" visualmente cosas que serían difíciles de explicar en palabras y funcionan para audiencias tanto técnicas como no técnicas, desde desarrolladores hasta directores. Esa es la razón por la que los DFD siguen siendo tan populares después de todos estos años.

Niveles de un DFD

Un DFD puede profundizar progresivamente en más detalle por medio de niveles y capas, concentrándose en una pieza en particular. Los niveles de un DFD se numeran 0, 1 o 2 y en ocasiones llegan incluso hasta el Nivel 3 o más. El nivel necesario de detalle depende del alcance de lo que estás tratando de lograr.

  • Nivel 0: Diagrama de contexto

En el diagrama de contexto se caracterizan todas las interacciones que realiza un sistema con su entorno (entidades externas), estas pueden ser otros sistemas, sectores internos a la organización, o factores externos a la misma. Se dibuja un sólo proceso que representa al sistema en cuestión y se escribe su nombre en dicha burbuja como un sustantivo común más adjetivos. De él solamente parten los flujos de datos que denotan las interrelaciones entre el sistema y sus agentes externos, no admitiéndose otros procesos ni almacenamientos en el dibujo, ya que estos son procesos estructurados y ordenados. Está diseñado para ser una vista rápida que muestra el sistema como un único proceso de nivel alto, con su relación con entidades externas.

  • Nivel 1: Diagrama de nivel superior

Brinda un desglose de piezas más detallado del diagrama a nivel de contexto. Se destacan las principales funciones que el sistema lleva a cabo.

  • Nivel 2: Diagrama de detalle o expansión

Va un paso más allá, ya que empieza a desglosar y analizar secciones específicas del diagrama de nivel 1. Cuanto más profundos sean los niveles, más texto tendrá el diagrama. Por eso, muchos diseñadores de sistemas prefieren no ir más allá del nivel 2

  • Nivel 3, 4, ...

Es posible el avance hacia los Niveles 3, 4 y más, pero ir más allá del Nivel 3 es poco usual. Hacerlo puede crear una complejidad que dificulte comunicar, comparar o modelar de forma efectiva.
